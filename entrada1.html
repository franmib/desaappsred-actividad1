Diseñando APIs REST con ASP.NET Core 8

Por: Ing. Carlos Mendoza

Especialista Principal de Backend en SOFTIA.ES

Con el lanzamiento de .NET 8, el desarrollo de APIs en C# ha alcanzado nuevos niveles de rendimiento y simplicidad. El concepto de Minimal APIs, introducido originalmente en la versión 6, se ha consolidado como la opción preferida para la creación de microservicios y servicios REST de alta escalabilidad gracias a su mínimo overhead.

1. ¿Qué son las Minimal APIs en ASP.NET Core?

Las Minimal APIs permiten declarar endpoints HTTP utilizando lambdas sencillas y métodos directos de mapeo sobre la instancia del servidor, reduciendo sustancialmente el código boilerplate que tradicionalmente conllevaban los controladores basados en clases (`ControllerBase`).

A continuación, se presenta el esquema estructural de una Minimal API típica configurada en .NET 8:

var builder = WebApplication.CreateBuilder(args);

                    // Configuración de la inyección de dependencias corporativa
                    builder.Services.AddScoped<IClientService, ClientService>();

                    var app = builder.Build();

                    // Endpoint Minimal API optimizado en C# 12
                    app.MapGet("/api/v1/clientes/{id:int}", async (int id, IClientService service) =>
                    {
                        var cliente = await service.GetByIdAsync(id);
                        return cliente is not null ? Results.Ok(cliente) : Results.NotFound();
                    });

                    app.Run();

2. Gestión Eficiente de Dependencias

En SOFTIA.ES hacemos mucho énfasis en no acoplar la infraestructura con el dominio de negocio. La inyección de dependencias (DI) integrada de ASP.NET Core maneja los ciclos de vida de los servicios mediante tres esquemas de resolución:

  • Transient: Creado cada vez que se solicita. Adecuado para clases ligeras y sin estado.
  • Scoped: Creado una vez por cada petición HTTP. Es la opción ideal para el uso de contextos de bases de datos (`DbContext`).
  • Singleton: Creado una única vez para toda la vida de la aplicación. Utilizado en caches en memoria estáticas o servicios de configuración compartidos.
